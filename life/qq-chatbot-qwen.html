NapCat · Qwen 7B · QQ 机器人 · 角色化对话

基于 NapCat + Qwen 7B 的本地群聊机器人

通过 GitHub 开源项目 NapCat 接入 QQ 协议，在裸 Qwen 7B 之上构建四层工程栈——身份注入、双轨记忆、质量守门、异步 API 化， 实现一个有个性、有记忆、输出可控的 QQ 群聊智能体。

🦙 Ollama + Qwen2.5 7B 🐧 NapCat QQ 接口 ⚡ asyncio 异步架构
System Prompt 工程 记忆系统 采样约束 WebSocket asyncio

🧱 四层工程架构总览

裸 Qwen 7B 是通用模型——不会角色扮演、没有记忆、输出不可控。通过四层增量叠加，将其改造为完整的群聊智能体：

#层级核心功能关键技术
1 身份层 角色人格注入 System Prompt + Few-shot 范例 + Recency Effect 锚定
2 记忆层 长期 + 短期双轨 滑动窗口 + 群聊摘要流水线 + 关键词检索 + 相似度去重
3 质量层 输出可控与防崩溃 采样参数约束 + 输出清洗 + 垃圾检测 + 自动降温度重试
4 工程层 API 化与异步并发 Ollama chat API + asyncio.Lock + WebSocket 长连接 + 多消息拆分
一句话总结：裸 Qwen 7B → 一个有身份、有记忆、输出可控、不会胡言乱语的 QQ 群聊机器人。

🎭 第一层：身份层 — 人设注入

裸模型是通用 AI，通过 System Prompt 注入完整角色人格。利用 7B 模型的 Recency Effect（最近信息权重最高），将人设规则放在 System Prompt 的最末尾，确保模型优先遵循角色约束。

👤 身份设定

男大学生、王者中路玩家、爱吐槽吃瓜看戏。

🗣️ 语言约束

不用句号逗号，换行代替停顿。0.5-1.2s 间隔连续发送。

💬 口头禅体系

有各种生活化的口头禅，不僵硬，有人情味。

😊 Emoji 情绪层

💀破防 😨震惊 😭哭泣 😲吃瓜。配合口头禅自然穿插，不过度使用。

Few-shot 范例设计

3 个正常聊天范例 + 2 个情绪爆发范例 = 5 个 Few-shot。告诉模型"平时自然说话，情绪到了才甩口头禅"，避免每句话都塞满梗的油腻感。正常聊天回复自然简短，情绪破防时才密集输出口头禅+emoji。

🧠 第二层：记忆层 — 长短期双轨

裸模型每次对话都是"失忆"状态。通过三层记忆架构让机器人知道"刚才在聊什么""这阵子群里发生了什么""这个人的偏好是什么"。

记忆类型存储结构存活时间作用
短期上下文 deque(maxlen=100) 按群隔离 最近 30 条拼入 Prompt 知道刚才在聊什么，多轮对话连贯
群聊摘要 每 100 条 → LLM 总结 → JSON 持久化 摘要攒 5 条 → LLM 浓缩 → 循环 知道这阵子群里的重要事件和话题走向
用户记忆 关键词检索 + 相似度去重（>70% 跳过） 持久化 JSON，跨会话 例：提到"火锅"→ 想起"A 爱吃火锅"

记忆流转流水线

📨 群聊消息 → 攒 100 条 → 🤖 LLM 总结
口语化 2-3 句 → 💾 摘要存盘 → 📊 攒 5 条 → 🤖 LLM 浓缩
1-2 段精华 → 🔄 循环

后台每 5 分钟扫描一次，异步执行，不阻塞消息处理。

🛡️ 第三层：质量层 — 防胡言乱语

裸 7B 模型输出不可控——格式崩溃、重复循环、回显 Prompt、乱码。通过三层防线兜底：

防线 1：采样约束

// Ollama API 采样参数 — 从源头控制输出质量 temperature: 0.75 // 降低随机性，减少天马行空 top_p: 0.9 // 截断低概率 token repeat_penalty: 1.15 // 防止陷入重复循环 stop: ["<|im_end|>", "<|im_start|>"] // 防 ChatML 标签泄露

防线 2：输出清洗

  • 物理强制无标点：逗号→空格，句号→换行，代码层面保证人设一致性
  • 非法字符过滤：正则保留中文+英文+数字+emoji+常用符号，剔除乱码
  • Prompt 回显检测：检测模型是否复读了 System Prompt 或指令片段

防线 3：垃圾检测 + 自动重试

is_garbage() 函数进行多维度判断：回显 Prompt 关键词、纯符号/数字、同字符重复 >15 次、高英文比例。检测到垃圾 → 降低温度（0.75 → 0.5）重试 → 还是垃圾 → 不发。宁可沉默，不崩人设。

⚙️ 第四层：工程层 — NapCat 接入 + API 化与异步

维度裸 Ollama本项目
调用方式 ollama run 子进程 aiohttp → /api/chat（原生 API，避免模板拼接 bug）
模板 手动拼 <|im_start|> → 双重嵌套 Ollama 原生 chat template

🚫 架构决策：为什么不使用专家路由网络

一个常见的设计思路是在 LLM 前加一层轻量分类器（路由网络 / MoE 门控）， 将不同意图的消息分发到不同模型或策略处理。经分析，该方案不适合当前场景，理由有三：

1. 延迟翻倍，群聊承受不起

✅ 当前方案

收到消息 → 1次 LLM 调用 → 回复

❌ 路由方案

收到消息 → 分类LLM → 回复LLM → 回复

Qwen 7B 本地推理一次耗时约 5–30 秒。两次调用意味着最差情况需要一分钟才回复， 群聊场景里消息早已刷过去。num_predict=5 救不了——编码阶段需要处理全部输入 token， decode 阶段短并不能抵消 encode 阶段的全部开销。

单次 LLM 调用已经是瓶颈，不能给它加排队。

2. Bot 的任务空间是一元的，不需要路由

专家路由适合的场景：

// 典型路由场景 — 多意图多能力 用户："帮我查今天的天气" → 路由到 [工具调用] 用户："这个函数怎么求导" → 路由到 [知识问答] 用户："哈哈哈笑死了" → 路由到 [闲聊]

但本 bot 只有一件事：扮演人物在群里聊天。不管是知识问答还是情绪宣泄， 都不需要换人设。

3. 已经有更轻量的决策层了

should_reply() 函数本质上就是一个零成本分类器：

// bot_main.py — 零延迟决策，不需要 LLM @机器人 / 触发词 → 必须回 群权限 off → 不回 其他 → 10% 概率随机插嘴

三行逻辑覆盖了全部路由需求，延迟为零。

什么时候才值得加路由

如果 bot 未来需要真正分裂人格、用不同模型处理不同任务：

// 未来扩展场景（当前不需要） [闲聊] → Qwen 7B + sxyl人设 [认真问答] → Qwen 14B + 专业prompt [生图] → 调 SD API

但目前既没有多模型，也没有多能力，加分类器纯属多一次 LLM 调用的延迟惩罚，换不来任何收益。

📊 技术价值总结

🎯 Prompt 工程

System Prompt 注入 + Few-shot 范例设计 + Recency Effect 利用。理解 7B 模型的注意力特性并据此优化 Prompt 结构。

🧠 记忆系统设计

短期滑动窗口 + 长期 JSON 持久化 + 关键词检索 + 字符集相似度去重。自主设计群聊摘要流水线（攒→总结→浓缩→循环）。

🛡️ 鲁棒性工程

多层输出质量保障：采样约束 + 物理清洗 + 垃圾检测 + 自动降温度重试。宁可沉默不崩人设。

⚡ 异步架构

aiohttp + asyncio.Lock + create_task 并发。WebSocket 长连接 + 后台定时任务（记忆压缩），全异步非阻塞。

🔌 系统集成

NapCat QQ 协议 → 本地 Ollama → Qwen 7B，完整端到端部署。群聊多用户上下文管理 + @触发 + 随机插话策略。
